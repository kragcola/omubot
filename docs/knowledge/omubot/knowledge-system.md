# Omubot 知识系统

## 记忆卡片、文档知识库和知识图谱的关系

Omubot 有三类长期上下文：记忆卡片、文档知识库、知识图谱。

记忆卡片是生产记忆的权威存储，保存用户偏好、群内事实、互动记录等和人或群相关的内容。文档知识库保存稳定资料，例如项目说明、群规则、Bot 操作手册和常见问题。知识图谱是从记忆卡片或文档片段派生出的实体关系事实，它是可重建的派生层，不替代记忆卡片或文档原文。

关键词：记忆卡片、文档知识库、知识图谱、ContextService、长期上下文

## 什么内容应该写进知识库

适合写进文档知识库的内容包括：项目架构、部署步骤、群规则、Bot 操作手册、固定设定、常见问题排查、稳定术语解释。

不适合写进文档知识库的内容包括：某个用户今天的临时状态、某个群刚刚发生的事件、一次对话中的短期上下文、用户私密偏好。这些内容应该进入记忆卡片或短期上下文。

关键词：知识库导入、稳定资料、群规则、操作手册、常见问题、用户记忆

## 知识库导入怎么写

知识库文档使用 Markdown。推荐一个主题一个文件，每个可独立回答的问题写成一个 `##` 二级标题。

好的知识块应该具备三个特征：标题包含用户会问的词，正文能独立回答问题，最后写出常见别名和关键词。

示例写法：文件标题可以是“Omubot 系统架构”，二级标题可以是“Omubot 的系统架构是什么”，正文直接回答“Omubot 使用 Kernel、Services、Plugins 三层架构”，最后补充关键词“Omubot 架构、三层架构、PluginBus”。

关键词：Markdown、二级标题、chunk、关键词、别名

## ContextPlugin 如何使用知识库

ContextPlugin 默认接管动态上下文。每轮真正准备回复时，它会调用 ContextService 检索记忆卡片、文档知识库和知识图谱事实，然后把命中结果合并成一个“上下文资料”块。

因此知识库不是工具调用，也不一定会在日志里出现 `tool_call`。如果要确认是否命中，应查看 `/admin/knowledge` 的上下文调试或 ContextPlugin 的 DEBUG 命中日志。

关键词：ContextPlugin、上下文资料、Prompt 注入、知识库触发、上下文调试

## 生产知识目录

生产聊天知识默认放在 `docs/knowledge`。研发文档、审计报告、路线图和计划文档继续保留在 `docs/wiki`、`docs/audits`、`docs/superpowers`，但默认不进入聊天知识库。

这样做可以避免 Bot 在回答日常问题时误召审计矩阵、开发计划、人设生成规则等不该直接进入对话的资料。

关键词：docs/knowledge、生产知识、研发资料、审计报告、误召回
