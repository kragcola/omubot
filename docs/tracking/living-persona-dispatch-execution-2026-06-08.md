# Living Persona 派单 — 角色生命化全量实施（执行追踪）

> 状态：2026-06-08 立。本文是 [Living Persona 系列](living-persona-part0-overview.md)（Part 0/A/B/C + [故事弧调研](living-persona-research-story-arc-2026-06-08.md)）的**执行派发版**，交付执行者按 Wave 顺序落地。
>
> 范围：把 omubot 从"无状态的当下生物"升级为"有情绪、有连贯生活、有伙伴的生命化角色"。本批覆盖 **A-M1（tension 单维）+ B-L1（角色驱动）+ B-L1.5（剧情弧账本）+ C-MVP（虚构伙伴卡）+ B-L2（反思）+ B-L3（重规划闭环）**。A 全维（M2–M4）、C 主体（真人入叙事）、SQLite 迁移 **搁置保留、不在本批**（见 §6 搁置区）。
>
> 配套（冲突时以**本文 §0 决策冻结表**为准，设计论证回溯各 part）：[Part 0 总纲](living-persona-part0-overview.md) §4 落地顺序、[Part A](living-persona-partA-dialogue-climate.md) §2 R5 决策、[Part B](living-persona-partB-generative-life.md) §4 L1–L3、[Part C](living-persona-partC-social-narrative.md) §5 红线 / §6.5 C-MVP、[故事弧调研](living-persona-research-story-arc-2026-06-08.md) §10 L1.5 清单。
>
> **执行原则（覆盖任何冲突项，逐条照做）：**
> 1. **Wave 严格串行**——下一 Wave 必须等上一 Wave「收口」全绿。唯一例外：**Wave 1N（B-L1）与 Wave 1E（A-M1）互不依赖，可并行**（Part 0 §4.1），但各自收口独立判定。其余 Wave 不允许跳、不允许抢跑。
> 2. **每 Wave 自带：D1 grep 证据 + 验证命令 + 回滚命令**——三者缺一不算闭环。动手前先跑 D1 grep，把实际行号/输出贴回本文对应回执位置。
> 3. **遇到与本文不符的现实（行号变了/文件不在/命令报错）→ 停，记录到 §5 偏差表，等验收人确认，不要自行猜着改。**
> 4. **全程灰度兜底**：每个落地层带开关、**默认关**、灰度对比；新路径未验证不夺旧路径。开关默认关 = 零行为变更，是每个 Wave 的回滚底线。
> 5. **本系列低优先级，不抢主线**（角色包补齐 + pmubot）。任何一 Wave 都可独立停在稳定态。
> 6. **每个落地层落地前确认回滚命令可用**（开关关 + `git checkout` + 删独立存储目录三选一可全退）。
> 7. **D5：跑全量 pytest 前先 `pkill -9 -f pytest`**，避免与 IDE 抢 sqlite 文件锁死锁。
> 8. **伦理红线（Part C §5，不可妥协）**：真人只能以"主观印象 / 共同经历"入叙事，**不可虚构其线下行为**；私聊内容不进群叙事。本批 C-MVP **只做虚构伙伴 `fiction`**，真人卡 `factual` 不在本批写入路径。
