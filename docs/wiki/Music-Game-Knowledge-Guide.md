# 音游知识库导入指南

本文说明 Omubot 当前知识库该如何导入音游相关资料，以及怎样写出更容易在群聊里命中的知识文档。

## 先说结论

如果你希望 Bot 在群聊中回答音游相关问题，推荐这样做：

1. 把生产知识写进 `docs/knowledge/music-games/`
2. 每个主题或游戏一个 Markdown 文件
3. 每个可独立回答的问题写成一个 `##` 二级标题
4. 在正文里显式写出中文名、英文名、常用缩写和别名
5. 导入后执行一次知识库重建索引
6. 先在 `/admin/knowledge` 搜索核对，再在群聊里 `@Bot` 提问

当前知识库是本地轻量 BM25/ngram 检索，不是 embedding。它对“标题词、别名词、正文关键词”非常敏感，所以文档结构比文采更重要。

## 推荐目录结构

建议在生产知识目录下新建：

```text
docs/knowledge/
  music-games/
    overview.md
    project-sekai.md
    arcaea.md
    maimai.md
```

说明：

- `overview.md` 负责跨游戏共通知识，例如“音游是什么”“常见术语是什么”
- 单独游戏一游戏一文件，避免多个作品混在一篇里稀释检索
- 不要把这类资料放进 `docs/wiki`、`docs/audits` 或 `docs/superpowers`，这些目录默认不进入生产聊天知识库

## 当前知识库最适合的写法

当前知识库按 `## 二级标题` 切 chunk，所以推荐模板如下：

```markdown
# 游戏名 / 英文名

## 这个游戏是什么

这里直接用 3 到 8 句话回答“它是什么、有什么特点、玩家通常怎么称呼它”。

关键词：中文名、英文名、缩写、别名

## 常见术语

这里直接解释玩家最常问的术语，尽量让一个 `##` 只回答一组相关问题。

关键词：术语A、术语B、术语C

## 常见问答

这里写最可能在群聊里被直接问到的问题，例如“rating 是什么”“箱活和混活有什么区别”。

关键词：问题关键词、常见说法、别称
```

为什么这么写：

- `#` 决定文档标题
- 每个 `##` 会变成一个独立 chunk
- 一个 chunk 应该尽量能“单独回答一个问题”
- “关键词”不是特殊语法，但能显著提升字面召回

## 音游文档的写作规则

### 1. 标题要像用户会问的话

不要只写：

```markdown
## 概述
```

更推荐：

```markdown
## Project SEKAI / pjsk / 世界计划 是什么
```

因为当前检索器非常吃标题词，标题里直接出现“pjsk”“世界计划”“Project SEKAI”，会比只写“概述”稳定得多。

### 2. 别名一定要在正文里落一遍

例如：

- `Project SEKAI`
- `世界计划`
- `pjsk`
- `sekai`

即使标题已经写过，正文里最好也自然重复一次。这样用户问缩写、英文名、中文名时都更容易命中。

### 3. 一个 chunk 不要塞太多主题

不推荐把“判定、编队、活动、角色、剧情、抽卡”全部塞进一个 `##`。  
更好的方式是拆成：

- `## 判定、FC、AP`
- `## 编队、属性和技能分`
- `## 活动、箱活、混活`

这样用户问具体问题时，召回更准，注入 Prompt 的上下文也更短。

### 4. 多写玩家真实会说的话

例如不要只写官方术语：

- `Potential`
- `Deluxe Score`

也要写玩家常说法：

- `ptt`
- `rating`
- `dx 分`
- `推分`
- `上分`

### 5. 不要把攻略楼、更新日志全文直接扔进去

这类内容通常：

- 太长
- 主题太散
- 包含大量时效信息
- 容易稀释召回

更适合的方式是先整理成“稳定问答版”知识，再导入知识库。

## 推荐的音游知识拆分方式

### 通用基础

适合放进 `overview.md`：

- 音游是什么
- 判定、连击、FC、AP 是什么
- 推分、定数、底力、段位 这类跨作品常见术语
- 群聊中如何向 Bot 提问音游问题

### 单独作品

每个游戏至少建议覆盖：

- 这款游戏是什么
- 常见别名和缩写
- 核心玩法
- 最容易被问到的 2 到 4 个术语
- 常见误解或易混淆点

## 导入后的操作步骤

### 1. 写入文档

把文档放到：

```text
docs/knowledge/music-games/
```

### 2. 重建索引

推荐二选一：

- 在 `/admin/knowledge` 点击“重建索引”
- 或重启 Bot

如果你修改的是容器内运行代码或 Docker 镜像内容，推荐直接：

```bash
docker compose build bot
docker compose up -d --no-deps --force-recreate bot
```

### 3. 搜索核对

先在 `/admin/knowledge` 搜索：

- `pjsk 是什么`
- `arcaea 天空音弧`
- `maimai rating`

确认能命中正确的 `source` 和 `title`。

### 4. 上下文调试

再在 `/admin/knowledge` 的“上下文调试”里输入：

- `pjsk 编队要看什么`
- `arcaea 的 ptt 是什么`
- `maimai dx 分和 rating 有什么关系`

确认命中结果里出现 `doc_chunk`。

### 5. 群聊实测

最后在群里优先用 `@Bot` 测试。  
原因不是知识库本身有问题，而是群聊里只有当 Bot 真正进入回复链路时，知识库才会被检索和注入。

## 常见失败原因

### 没有 `##`

如果文档只有：

```markdown
# 标题
正文正文正文
```

那当前实现很可能切不出稳定 chunk，结果就是“文件在目录里，但搜索不到”。

### 标题太抽象

像 `## 概述`、`## 杂谈`、`## 其他` 这种标题，对检索帮助很小。  
更推荐把用户会问的词直接写进标题。

### 只写中文，不写英文或缩写

很多群聊会混着说：

- `pjsk`
- `sekai`
- `ptt`
- `dx`
- `fc`
- `ap`

如果文档里完全没有这些词，召回会明显变差。

### 导入后没重建索引

当前知识库不会因为你新加了 Markdown 就每轮自动重扫，所以文档写完后一定要重建索引或重启。

## 一套最低可用模板

如果你要快速导入一款新音游，最低可以这样写：

```markdown
# 游戏名 / 英文名

## 游戏名 / 缩写 / 别名 是什么

这里回答它是什么、核心特点是什么、玩家通常怎么叫它。

关键词：中文名、英文名、缩写、别名

## 核心玩法

这里回答最关键的玩法机制。

关键词：核心机制、玩法关键词

## 常见术语

这里解释玩家高频会问的术语。

关键词：术语1、术语2、术语3
```
