# 轻量语义检索

Omubot 的语义增强当前坚持“默认轻量、可安全降级”的路线，不把 embedding、FAISS、向量数据库塞进默认 Docker。

当前落地范围主要在两处：

- `services/memory/retrieval.py`：记忆卡片的轻量语义匹配
- `services/similarity.py`：统一的 `SimilarityProvider` 抽象

## 目标

要解决的问题不是“做一个重量级向量平台”，而是补上关键词检索的一个明显盲区：

- 群友说的话和记忆卡片含义相近
- 但不是完全相同的字面词
- 这时 `LIKE '%关键词%'` 可能搜不到

例如：

- 卡片：`对猫毛过敏`
- 当前对话：`对猫会过敏`

这类场景下，轻量语义匹配会在关键词失败后补一次兜底。

## 当前行为

`RetrievalGate` 现在仍保持原有四层主结构：

1. 新会话：全量注入记忆
2. 周期刷新：定期全量注入
3. 关键词匹配：优先走已有搜索
4. 最小提示：提醒模型可用 `lookup_cards`

当且仅当满足下面条件时，才会走语义分支：

- `memory.semantic.enabled = true`
- 当前轮有 `conversation_text`
- 关键词匹配没有命中卡片

命中后会返回带 `轻量语义匹配` 标记的记忆块。

## 后端选择

### `ngram`

默认后端，当前主实现。

特点：

- 无额外依赖
- 可预测
- 适合短文本、缩写、轻量模糊匹配
- 适合默认 Docker

局限：

- 主要依赖字符重合
- 对真正的“同义表达”能力有限
- 不能替代 embedding 检索

### `embedding`

目前只是 optional extra 预留位。

当前默认行为：

- 如果配置成 `embedding` 但未安装对应实现
- 运行时会自动回退到 `ngram`
- 会记录 fallback / error 指标
- 不会因为后端缺失而中断对话

这也是目前推荐的安全策略：先让配置层和调用点稳定，再决定是否引入重依赖。

## 系统页状态

系统页服务健康中的 `Memory` 项会带出语义检索状态：

- 是否启用
- 请求后端
- 实际生效后端
- 查询次数、命中次数、命中率
- fallback / error 情况与最近错误

因此即使未来启用了 `embedding` optional extra，也能第一时间看出是不是已经退回 `ngram`。

## 为什么不默认上 embedding

原因不是“embedding 没价值”，而是当前阶段优先级不同：

- Omubot 当前更强调陪伴型 QQ 群 bot 的轻量常驻
- 默认镜像越轻，部署、重建、排查越稳
- 向量依赖一旦进默认栈，会立刻抬高镜像、冷启动、依赖冲突和维护成本

所以路线图里的策略一直是：

- v3 / Phase 5：先把抽象、调用点、降级策略和状态面板补齐
- v3.5 以后：再评估 optional extra 的 embedding 实装

## 后续可扩展方向

下一步如果继续推进，可以沿这条线走：

1. 为 `EmbeddingSimilarityProvider` 增加真实实现，但只在 optional extra 安装后启用。
2. 给黑话系统的释义相似度、漂移判断和候选合并复用同一套 provider。
3. 继续把 semantic metrics 从系统页扩展到更细粒度的管理视图，例如按会话或按群统计。
4. 继续保持“未安装重依赖也能完整运行”的默认体验。
